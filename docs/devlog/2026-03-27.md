## 1. 群组消息与私聊分离

**问题**: 群组消息和私聊消息混在一起，私聊标签能看到群组消息，群组标签看不到任何内容。

**根因**: `Channel::post_group()` 先调用 `post_message()`（event 立即广播，`group: None`），然后才在存储的消息上设 `group` 字段。TUI 收到的 event 里 `group` 永远是 `None`。

**修复**:
- 重写 `post_group()` — 在构造 Message 时就设好 `group` 字段，一次性 push + emit event
- DM 标签过滤加 `if line.group.is_some() { return false; }` 排除群组消息
- 群组标签 streaming/thinking 预览修复 — 新增 `group_members` 字段做成员匹配
- 系统标签群组消息加紫色 `[群组名]` 标记

**文件**: `channel.rs`, `messages.rs`, `app.rs`
**测试**: 新增 8 个测试（7 TUI 过滤 + 1 event group 字段）

## 2. 群组模式用户输入修复

**问题**: 用户在群组标签输入消息，发送给了 main agent 而不是群组。

**根因**: `handle_input()` 没有处理群组模式，所有输入都走 DM 路径。

**修复**: 在 `handle_input` 开头加群组模式分支，走 `post_group` + `dispatch_group_sequential`。

**文件**: `app.rs`

## 3. 群组上下文 @mention 泄漏

**问题**: agent 在群组回复中 @mention 其他 agent 时，消息走 directed routing 泄漏到私聊。

**修复**: 群组上下文中（`group.is_some()`），始终作为群组消息发出，跳过 @mention 路由。

**文件**: `app/mod.rs` (do_prompt_inner 完成处理器)

## 5. GroupMessage handler 缺少 has_bus_activity

**问题**: agent 通过 `bus_group_message` 工具发消息后，`has_bus_activity` 没设为 true，可能触发重复自动回复。

**修复**: 在 `post_group` 之后加 `agent.has_bus_activity = true`。

**文件**: `app.rs`

## 7. 群组讨论顺序 dispatch + 上下文积累

**问题**: 所有 agent 同时收到群组 prompt 并行回复，没有对话历史，导致重复/矛盾/混乱。

**修复**:
- 新增 `Channel::group_history()` — 返回群组最近 N 条消息
- 新增 `format_group_prompt()` — 构建带历史上下文的 prompt
- 新增 `dispatch_group_sequential()` — 逐个 dispatch，每个 agent 完成后其回复加入下一个的上下文
- Agent A 只看历史 → Agent B 看历史+A → Agent C 看历史+A+B

**文件**: `channel.rs`, `app.rs`
**测试**: 新增 3 个 group_history 测试

## 10. app.rs 模块拆分

**问题**: `app.rs` 有 2,355 行，占 TUI crate 的 54%，所有逻辑堆在一起。

**修复**: 拆分为 5 个子模块，每步做 RGR：

| 步骤 | 操作 | 行数 |
|------|------|------|
| Step 1 | app.rs → app/mod.rs | 0 改动 |
| Step 2 | 提取 bus_events.rs | ~475 行 |
| Step 3 | 提取 commands.rs | ~282 行 |
| Step 4 | 提取 prompting.rs | ~596 行 |
| Step 5 | 提取 lifecycle.rs（合并重复 event listener） | ~270 行 |

**结果**: mod.rs 从 2,355 行降到 683 行。最大文件 683 行，无文件超 800 行。

**文件**: `app/mod.rs`, `app/bus_events.rs`, `app/commands.rs`, `app/prompting.rs`, `app/lifecycle.rs`
