## 会话 2 — 待修复问题（从日志分析）

### 问题 1：main 不委派，自己干活

**日志证据**（20260329-071617.md）：
- 07:16:38 用户："帮忙去多方面调查下雷石这家公司"
- 07:17:11 main **自己做了 web 搜索**，返回了完整调研报告（耗时 33 秒）
- 07:19:29 main **又自己搜索了第二轮**（耗时 2 分 18 秒）
- 07:21:28 用户："不是多agent吗，怎么变成你一个人了"
- 07:22:01 main 才创建了 3 个 agent

**根因**：main 的提示词虽然写了 "delegate complex tasks"，但 main 还是选择自己做。提示词约束力不够强——"Complex tasks (need multiple agents)" 这个判断条件太模糊，main 认为自己能搜索就搜索了。

**修复方向**：
- 提示词要更强制：收到用户任务后**第一步永远是创建 agent 委派**，除非是简单的一句话问答
- 或者：去掉 main 的 WebSearch 工具能力，强制它只能通过 agent 搜索

### 问题 2：agent 创建后没收到任务

**日志证据**：
- 07:21:44~07:21:59 三个 agent 陆续上线
- 07:22:01 main 说"他们会各自调研完后向我汇报"
- **日志到此结束，没有任何 agent 活动**

**根因**：main 创建了 agent 但可能没有用 `bus_send_message` 给他们发任务。或者 main 在 `bus_create_agent` 时指定了 `task` 参数但 agent 连接太慢、任务丢了。

**需要验证**：
- `bus_create_agent` 的 `task` 参数是否正确触发了 `do_prompt`
- agent 连接后是否收到了任务
- 是否有 "未连接超时" 日志

### 问题 3：main 响应太慢

- 第一次回复：33 秒（做了 web 搜索）
- 第二次回复：2 分 18 秒（做了更多搜索）
- 用户期望：几秒内回复 + 立即委派

**根因**：同问题 1，main 自己干活而不是委派。

### 问题 4：main-b 从未被使用

日志显示 main-b 上线后再无活动。所有用户消息都发给了 main，main 不忙所以 main-b 从未接活。

**这其实是正确行为**：如果 main 能快速委派并返回，main-b 就不需要。问题是 main 太慢才导致需要 main-b。
